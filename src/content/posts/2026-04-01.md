### Andrej Karpathy — 前 Tesla AI 总监，OpenAI 创始团队成员 #话题/前端xAI

**原文：**
> New supply chain attack this time for npm axios, the most popular HTTP client library with 300M weekly downloads. Scanning my system I found a use imported from googleworkspace/cli from a few days ago. The installed version (luckily) resolved to an unaffected 1.13.5, but the project dependency is not pinned, meaning that if I did this earlier today the code would have resolved to latest and I'd be pwned. I think ultimately the defaults of package management projects (pip, npm etc) have to change so that a single infection does not spread through users at random and at scale via unpinned dependencies.

**中文解读：**
npm 生态又遭供应链攻击，这次是 axios，周下载量 3 亿的 HTTP 客户端库。Karpathy 自己差点中招：他用的 googleworkspace/cli 依赖了 axios 但没锁版本，幸亏安装时间点稍早，解析到了未受影响的 1.13.5。他的核心主张是：包管理工具的默认行为必须改变，不锁版本的依赖解析方式让一次投毒就能大规模扩散。

**大白话：**
有人往 npm 上最火的网络请求库里投了毒。如果你的项目没锁版本号，可能不知不觉就中招了。Karpathy 说问题出在 npm 的默认设置太宽松。

**辩证视角：**
🟢 如果他说得对：包管理工具应该默认锁版本、设置发布年龄约束，这会显著降低供应链攻击面。对前端团队来说，现在就应该审查 lockfile 和 CI 中的依赖解析策略。
🔴 但另一面是：严格锁版本会降低安全补丁的自动流入速度。2022 年 Log4Shell 之所以危害巨大，恰恰因为太多项目锁着旧版本不更新。安全是一个"开放 vs 锁定"的永恒张力，没有银弹。
🎯 判断线索：关注 npm 官方是否会在 2026 年内调整默认的版本解析策略（如引入 release-age 约束作为 opt-out 而非 opt-in）。

**中文解读：**

**大白话：**
